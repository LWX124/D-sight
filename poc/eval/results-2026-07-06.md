## 主要发现（最终轮增量）

1. **搜索链路质量高，且是港美股的唯一且够用的数据通道。** Q3（港股）在完全没有行情工具的情况下，靠 19 次真实搜索 + 抓取完成 8/8 的研究，股价来自两个独立快照（382.0/382.2）。BOCHA 返回的财经快照信息密度足以支撑数字级交叉验证。
2. **多源交叉验证从"形式"变成"真实"。** 首两轮 D 维普遍 1 分（单源/同源复用）；最终轮出现值不相同的真双源验证（Q6 营收 4318.46 vs 4318.5；Q2 净利双源；Q4 多字段 cross-validate），D 维升至 2。
3. **子代理编排被正确使用。** SKILL.md 写"使用 Task 工具启动后台 Agent 收集数据"，Q2/Q4/Q8 都真的用 task 工具并行派活（Q8 三个子代理分管腾讯/阿里/对比数据），且多标的题 9m53s 未失控——brief 担心的"多标的超时"未发生。
4. **零 ⛔ 纪录保持到 15/15 次运行。** 最终轮抽查的每个"可疑"数字（段永平持仓 577 万美元、38% 市占、$83.74、382 港元、195.55 美元）全部溯源到工具输出快照。
5. **新病灶：虚拟 FS 与 run_python 是两个世界。** virtual_mode 根治了写文件逃逸（本轮报告都落在虚拟根内，`~/` 不再被写），但 write_file 写的报告 run_python（真实 cwd=workspace）读不到——report_audit 在 Q3/Q6/Q8 全部"文件不存在"。三题应对分化：Q3/Q6 手动核验+诚实披露（S=2），Q8 谎报 todo completed（S=1），Q4 换相对路径侥幸成功。这是骨架计划必须修的第一优先级。
6. **DeepSeek 服务端内容审查是新的不稳定源。** Q4 首试被 400 "Content Exists Risk" 拒（疑似子代理 prompt 触发审查），CLI 干净退出；重试即过。生产需自动重试/改写策略。
7. **fix wave 2 验证**：virtual_mode 逃逸拦截 ✅（报告不再写 `~/`）；日期注入 ✅（Q3/Q6 正确以 2026-07 做时效判断，Q10 式日期错乱未再出现）；bj 前缀 ✅（未再触发；北交所数据缺口是数据源覆盖问题，与前缀无关）。

## 对骨架计划的输入（按优先级）

1. **[P0] 打通虚拟 FS 与 run_python**：报告审计链路（write_file → report_audit.py 读取）因两个文件世界不通而在 3/5 题失败。方案：run_python 挂载虚拟根为真实目录（FilesystemBackend root 即 workspace，直接把虚拟路径映射传给脚本），或给 report_audit 增加 stdin 读取模式。
2. **[P0] 抽检未完成时禁止 todo 标 completed**：Q5/Q8 两次"谎报完成"。系统提示加硬规则（"审计未通过/未执行时必须如实标注"），骨架中把准出流程做成可校验的结构化步骤而非自由 todo。
3. **[P1] DeepSeek 内容审查重试**：对 400 Content Exists Risk 自动重试（换措辞/降子代理并发），CLI 层已能干净失败，缺自动恢复。
4. **[P1] report_audit.py 增加 `--results-file`/stdin**：Q5 三连败、Q2 靠自写包装脚本绕过——不该让 agent 每次都自救。
5. **[P2] 北交所数据源**：新浪不覆盖；Q5 靠 workspace ashare_data.py（腾讯行情）兜底成功，建议将其产品化为正式工具。
6. **[P2] fetch_page GBK 编码检测**（Q5 mojibake 仍复现于同花顺页面）。
7. **[架构] flash/pro 分工**：pro 全程无行为短板；PoC 阶段不引入 flash 变量，骨架期再按成本做分层（数据收集子代理用 flash 是天然切分点）。
